## Architectural Rule

Hermes agents generated by `/agentfactory` must be role-scoped execution shells. REVCLI/Revis owns policy, approvals, workflow state, and audit. Odoo or the configured database owns business truth. Hermes must never become the source of truth for permissions, commercial state, customer-visible sends, or CRM writes.

## Authority Chain

| Layer | Owner | Allowed Hermes Interaction | Forbidden Hermes Interaction |
|-------|-------|----------------------------|------------------------------|
| Hermes agent | Generated runtime profile | Interpret operator intent, draft, classify, call allowed runtime actions, escalate, log evidence. | Directly mutate CRM/DB, approve its own risky action, hide credentials, bypass REVCLI workflows. |
| REVCLI headless pilot | Domain service and workflow authorization policy | Execute domain actions through allowed roles and scopes. | Accept unmanaged direct writes without action authorization. |
| Revis SaaS | Tenant/profile/workflow/approval/runtime event control plane | Assign profiles, govern approvals, record runtime events, export evidence, suspend execution. | Treat demo/local memory as production proof. |
| Odoo/CRM/Postgres | System of record | Store mediated business state and recovery evidence. | Receive direct Hermes writes when a governed adapter/action exists. |

## Required Runtime Evidence For REVCLI-Ready Agents

Every REVCLI-facing Hermes agent must generate these artifacts before it can be more than `experimental`:

| Artifact | Required Evidence |
|----------|-------------------|
| `hermes.manifest.md` | Role, tenant/team scope, system-of-record boundary, action authority matrix, credential strategy, egress policy, audit contract. |
| `hermes.runtime.json` | Entry point, cwd, argv allowlist, config allowlist, env allowlist, input schema, max input size, allowed/denied runtime actions, approval gates, audit sink, kill switch, fixtures. |
| `hermes.verify.md` | Executable smoke, boundary, approval, argument constraint, audit, durable resume, and kill-switch tests with evidence paths. |
| `hermes.kill-switch.md` | Runtime-backed disable command or procedure, expected disabled status, expected audit event, last test result. |
| `hermes-registry.md` row | Verify link, kill-switch link, runtime evidence link, verification isolation, last kill test. |

## Lowest-Risk First Agent

`revis-opening-seller-copilot` is the safest first production candidate because it can be constrained to `read-only` plus draft generation. It helps reps triage opportunities and prepare outreach/reply decisions without sending messages, mutating CRM state directly, changing owners, pricing, closing outcomes, or touching credentials.

`revcli-nurture-signal-monitor` is the canonical slug for the highest-leverage first automation candidate because it matches the existing Agent Factory blueprint. It must stay `experimental` until REVCLI exposes a runtime-backed kill switch and audit path for the ingest/trigger command.

## REVCLI Runtime Blocks AgentFactory Must Respect

| Block | Reason | AgentFactory Response |
|-------|--------|-----------------------|
| Missing runtime-backed kill switch | A documented env var is not enough if the entrypoint still mutates state. | Generated agent remains `experimental` or `blocked`; verification must prove no domain-service writes occur while disabled. |
| Missing runtime audit sink call | CRM activity logs are not equivalent to runtime trace/audit evidence. | Verification must inspect runtime audit/event evidence, not stdout alone. |
| Direct Odoo/CRM access requested | REVCLI owns policy and adapter semantics. | Remove direct write grant or block as runtime bypass. |
| Arbitrary command args/config path | An exact binary can still run in unsafe modes. | Require argv allowlist, config allowlist, input schema, max input size, and negative tests. |
| Unknown workflow risk | REVCLI defaults unknown workflows toward approval, but production agents need explicit catalog proof. | Treat as approval-required and keep status experimental until catalog/policy evidence exists. |
| Demo/local runtime proof only | Demo/local adapters are not production evidence. | Allow draft/experimental only; require production-like runtime smoke before active. |

## REVCLI AgentFactory Overlay Checklist

- [ ] Agent declares one primary REVCLI role or justified extension.
- [ ] `hermes.runtime.json` names REVCLI/Revis as policy authority when target runtime is REVCLI-facing.
- [ ] System-of-record writes are mediated by REVCLI domain service, runtime API, MCP bridge, or approved adapter.
- [ ] Every side-effecting action has an approval gate, audit event, idempotency key, and rollback or compensation path.
- [ ] First outbound, proposal send, close won/lost, owner reassignment, pricing/redline, evidence export, and profile suspension are approval-gated.
- [ ] Runtime command grants include cwd, argv allowlist, config allowlist, env allowlist, input schema, max input size, and denied flags.
- [ ] Runtime evidence includes trace ID, audit sink event, workflow run ID, approval ID when applicable, and business object ID.
- [ ] Kill switch proves the disabled path prevents new work and writes an audit event.
- [ ] Sales/prospecting agents terminate records into `closed-won`, `closed-lost`, `nurture-active`, or `archived`.
- [ ] No production execution uses WhatsApp, Telegram, personal Signal, SMS, or personal email as the governed action channel.
